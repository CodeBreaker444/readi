---
title: "System Status Rules"
description: "How OPERATIONAL, MAINTENANCE, and NOT_OPERATIONAL are computed on every fleet load."
---

A system status is **not stored**. It is computed fresh every time the fleet list loads.

---

## Computation Logic

```
For each system (tool):

  1. Check components:
     Find any active component (component_active = Y)
     where expiration_date <= today
     → If found: NOT_OPERATIONAL

  2. Check maintenance tickets:
     Find any ticket where ticket_status != CLOSED
     → If found: MAINTENANCE

  3. Neither condition → OPERATIONAL
```

**Priority:** `NOT_OPERATIONAL` beats `MAINTENANCE`.

---

## Status Definitions

| Status | Trigger |
|---|---|
| `OPERATIONAL` | No expired active components, no open tickets |
| `MAINTENANCE` | `ticket_status IN (open, in_progress, resolved)` |
| `NOT_OPERATIONAL` | `component_active = Y` AND `expiration_date <= today` |

<Note>
Set `component_active = N` on an expired component to remove it from the status computation without deleting it.
</Note>

---

## Where Status Appears

| Feature | Effect |
|---|---|
| Fleet list | Status badge per drone |
| Operations Board | Status change blocked if NOT_OPERATIONAL or MAINTENANCE |
| Mission creation | Non-operational drones greyed out and unselectable |
| Operation Calendar | Missions on non-operational systems shown with strikethrough |
| Document Repository | Documents linked to non-operational systems flagged |
| Logbook filter | Drone status shown per entry |
| AI Agent | DECOMMISSIONED intent queries `tool_component.expiration_date` |

---

## Clearing Each Status

| Status | How to clear |
|---|---|
| `NOT_OPERATIONAL` | Set `component_active = N` on the expired component, or update `expiration_date` after replacement |
| `MAINTENANCE` | Close the maintenance ticket (`ticket_status = CLOSED`) |

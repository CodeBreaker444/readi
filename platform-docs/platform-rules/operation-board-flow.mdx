The Operations Board is a Kanban-style view where mission status changes are performed by dragging cards between columns. This page documents the exact sequence of events.

---

## Status Transitions Supported

| From | To | action_type in transaction_signs |
|---|---|---|
| `PLANNED` | `IN_PROGRESS` | `mission_start` |
| `IN_PROGRESS` | `COMPLETED` | `mission_complete` |
| Any non-final | `CANCELLED` | `mission_cancel` |
| `IN_PROGRESS` | `PLANNED` (revert) | `mission_revert` |

---

## What Gets Recorded

Every successful drag creates two records:

1. **`transaction_signs`** â€” cryptographically signed record of who authorised the change, when, and to what state
2. **Audit log** â€” plain-text event log entry: `"Mission status changed from X to Y by [user]"`

These two records together make every status transition fully traceable and tamper-evident.

## Error States

| Error | Cause | User sees |
|---|---|---|
| Authorization PIN not set up | No key pair in `user_authorization_keys` | Modal error: "Authorization PIN not set up. Go to Settings â†’ Security." |
| Wrong PIN | Client-side decryption fails | Modal: "Incorrect PIN. Please try again." |
| Daily declaration missing | No declaration filed today for this pilot | Board action blocked: "Pilot has not filed a daily declaration" |
| System not operational | `NOT_OPERATIONAL` or `MAINTENANCE` status | Board action blocked: system status warning shown |
| Network error | API call fails | Card snaps back to original column, toast error shown |

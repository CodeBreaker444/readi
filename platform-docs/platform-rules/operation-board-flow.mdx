title: "Operation Board Flow"
description: "Step-by-step sequence when a mission card is dragged on the Operations Board."
---

## Full Drag Sequence

```
1. User drags mission card from column A to column B
2. targetStatus determined from column B name
3. Pre-checks: system operational? status transition valid?
4. Authorization check triggered
   → Authorization modal opens
   → User enters PIN
   → JWT signed, POST /api/authorization/transaction-sign
   → transaction_sign_id returned
5. POST /api/operation/board/status
   { missionId, targetStatus, transactionSignId }
6. Server checks:
   a. Operations view permission check
   b. Daily declaration check
   c. System operational status check
   d. Mission status updated in database
7. pilot_mission.fk_mission_status_id updated
8. Card renders in new column, notification + audit log created
```

---

## Status Transitions

| From | To | `action_type` |
|---|---|---|
| `PLANNED` | `IN_PROGRESS` | `mission_start` |
| `IN_PROGRESS` | `COMPLETED` | `mission_complete` |
| Any non-final | `CANCELLED` | `mission_cancel` |
| `IN_PROGRESS` | `PLANNED` | `mission_revert` |

---

## What Gets Recorded

Every successful drag creates two records:

1. **`transaction_signs`** — signed record of who authorised the change
2. **Audit log** — plain-text: "Mission status changed from X to Y by [user]"

---

## Error States

| Error | Cause | Result |
|---|---|---|
| PIN not set up | No `user_authorization_keys` entry | Modal: "Authorization PIN not set up. Go to Settings → Security." |
| Wrong PIN | Client-side decryption fails | Modal: "Incorrect PIN. Please try again." |
| No daily declaration | No declaration filed today | Board action blocked |
| System not operational | NOT_OPERATIONAL or MAINTENANCE | Board action blocked |
| Network error | API call fails | Card snaps back to original column |

## What's New

### Non-Operational Systems — Visual Warnings Across the App

We've made it much clearer when a system or drone is no longer operational due to an expired component. Instead of having to check manually, the app now shows these warnings automatically wherever it matters.

**Operation Calendar**
Missions linked to a non-operational system appear grayed out with a strikethrough on the calendar, so you can immediately spot which scheduled operations are affected.

**Document Repository**
Documents linked to a non-operational system are now shown grayed out in the list. A small "Not Operational" label appears next to the document name so you always know at a glance. When uploading a new document, if the selected system is non-operational, a warning banner is shown inside the form — the document can still be saved, but you'll know the system can't be used for new missions.

**Mission Creation**
When creating a new operation, drones belonging to a non-operational system are grayed out and can no longer be selected. If you're editing an existing mission that already has a non-operational drone assigned, a clear warning banner appears to let you know that the system has expired and the mission would be blocked.

---

### Drone Classes — Now Independent Per Component

Previously, drone classes were managed from a shared global list. If you deleted a class from the global list, it would disappear from every component that had it selected — which was not the intended behaviour.

Now, each drone component keeps its own list of classes independently:

- Classes you assign to a component belong to **that component only**
- Removing a class from one component does not affect any other component
- You can still pick from the global suggestions list as a starting point, or type in a completely custom class value that isn't in the global list at all
- The global class manager is now just a pool of suggestions — it no longer controls what individual components have assigned

## What's New

### Component Expiry — Flight Hours Type Added

The **Expiry Type** dropdown on the Add and Edit Component forms now includes a **Flight Hours** option alongside the existing Expiration Date, Flights, and Mixed types.

- **Flight Hours** — decommissions the component once a specified number of flight hours is reached. Accepts float values up to 9999 (e.g. 500.5).
- **Mixed** — now supports all three limits: expiration date, flight count, and flight hours. The component is decommissioned as soon as any one of the configured limits is reached.

---

### Audit Log — Event Details Modal

Every row in the Audit Log now has a **Details** button that opens a modal with the full context of the event:

- Full description text
- Section and Entity ID
- Exact timestamp and IP address
- User name, email, and role
- Raw metadata (or structured DCC details for DCC bug reports)

---

### PDF Exports Now Use Compression

All PDF files generated by the app are now compressed at the point of creation. This applies to:

- **Table exports** across any page that has an Export button (operations, logbook, clients, etc.)
- **Mission reports** generated from the operation detail view

File sizes are noticeably smaller with no change to content or layout. Downloads are faster, and storage is reduced both locally and when sending reports to clients.

---

### Company Management — Activate / Deactivate Toggle

The company list page and the company detail page now show an **Activate** button when a company is currently disabled, instead of always showing **Deactivate**.

- On the company list, each row shows either a red **Deactivate** or a green **Activate** button depending on the company's current status.
- On the company detail page, the header button toggles between **Activate Company** (green) and **Deactivate Company** (red) based on the current state.
- Activating a company takes effect immediately — no confirmation dialog is required since the action is non-destructive.

---

### Warehouse — Parent Component Cascade

When a parent component is moved to the warehouse, all child components attached to it are automatically warehoused at the same time.

Previously, sending a parent to the warehouse left its children attached to the system with no parent — they had to be moved individually. Now a single detach operation cleans up the full assembly in one step.

---

### System Status — Dismissed

A new **Dismissed** status can be set on any system. It behaves like Not Operational for mission planning:

- Dismissed systems are **greyed out and disabled** in the drone selection dropdown when creating a mission.
- A warning banner is shown if the selected system is Dismissed.
- The status is shown as a **DISMISSED** tag in the dropdown, distinct from the red NOT OPERATIONAL tag.
- Priority in the fleet status display: `NOT_OPERATIONAL > DISMISSED > MAINTENANCE > OPERATIONAL`.

Unlike Not Operational (which is computed automatically when a component expires), Dismissed is set manually to administratively remove a system from active use.

## Bug Fixes

### Maintenance Ticket — Component Dropdown Now Shows Component Code

When opening a new maintenance ticket and selecting a component, the dropdown was showing the component type and serial number (e.g. `CAMERA — SN12345`) instead of the component's own code (e.g. `CAM-001`).

The component code is now displayed, making it easy to identify the correct component when a system has multiple components of the same type.

## What's New

### Supabase Client → Prisma ORM Migration

All database operations that previously went through the Supabase JS client have been migrated to Prisma ORM. This affects queries, mutations, and real-time data lookups across the platform.

Key improvements from the migration:

- **Type safety** — Prisma generates types directly from the database schema, so query results are fully typed end-to-end with no manual casting.
- **Consistent error handling** — Prisma surfaces structured error codes (e.g. `P2025` for record-not-found, `P2003` for FK constraint violations) that map cleanly to API responses.
- **Transaction support** — Multi-step writes (create + update, null FK + delete) can now be wrapped in atomic transactions, preventing partial state on failure.
- **Schema-driven** — The Prisma schema (`prisma/schema.prisma`) is the single source of truth for table structure, relations, and cascade rules — no more divergence between the ORM model and the actual database.

Supabase Auth is still used for session handling; only the database query layer has changed.

---

### Admin Route Protection — Company Directory Now Returns 403

Admin users who navigated directly to `/company` previously saw the company directory UI with no data instead of an access-denied page. This was caused by two permission shortcuts in `hasRoutePermission` and `canAccessRoute` that unconditionally returned `true` for the ADMIN role, bypassing the actual permission check.

- Both shortcuts have been removed. ADMIN is now evaluated against its declared permission set like every other role.
- The `add_company` permission (which covers `/company` routes) has been removed from the ADMIN permission set — company management is a SUPERADMIN-only operation.
- Navigating to `/company` as an Admin now correctly redirects to the `/unauthorized` page.

## Bug Fixes

### API Key Deletion Failed with Foreign-Key Constraint

**Issue:** Deleting an API key from Settings > Security threw a Prisma error when any flight request had previously been submitted using that key. The `flight_requests` table holds a foreign key to `api_keys` with `onDelete: NoAction`, so the database rejected the delete.

**Fix:** Before deleting the key, `fk_api_key_id` is now set to `null` on all related flight requests. The key is then deleted cleanly.

---

### Assigned Client Not Shown in Edit User Modal

**Issue:** When opening the edit user modal for a user with an assigned client, the client dropdown was always blank. The client ID was being read from the database but was silently dropped before the user list was returned to the frontend.

**Fix:** `fk_client_id` is now included in the formatted user object returned by `getUserListByOwner`, so the edit modal correctly pre-selects the assigned client.

---

### Prisma Update Error When Saving Client Assignment

**Issue:** Saving a user edit as a SUPERADMIN resulted in `"No record was found for an update"`. SUPERADMIN sessions carry `ownerId = 0`, and the Prisma WHERE clause `{ user_id: X, fk_owner_id: 0 }` matched no rows because no users have `fk_owner_id = 0` in the database.

**Fix:** The `fk_owner_id` condition in the update WHERE clause is now only applied when `owner_id > 0`, allowing SUPERADMIN to update any user regardless of owner scope.

---

### Org Chart Edit Modal — Always Defaulted to Company Root as Parent

**Issue:** Opening the Edit Node modal on any org chart node always pre-selected "Company Root" in the Reports To dropdown, even when the node was a direct child of another member.

**Fix:** The modal now searches the flat members list for whichever member has the current node in its `children` array, and pre-populates the dropdown with the actual parent. Nodes that genuinely report to the company root continue to show "Company Root".

## What's New

### Actual Start Time on Post Flight

The Post Flight tab in Edit Mission now shows and lets you edit **Actual Start**, alongside Actual End. Previously only the end time was tracked, so an imported flight's takeoff time had nowhere to go.

### Update Password for Clients & Users

The Client Management and Team Management screens now include an **Update Password** action for accounts that haven't been activated yet.

- **Where to Find It** — On the Clients table, open the row menu for any pending client and choose "Update Password." On the Personnel table, a new key icon appears next to "Resend Invite" for any pending user.
- **Pending-Only Guard** — The action is only available while the account is still awaiting activation. Once a client or user has logged in and completed activation, the option disappears and the password can no longer be reset this way, preventing accidental overwrites of a credential the account holder has already chosen.
- **Automatic Email Delivery** — Submitting a new password immediately updates the account's credentials and emails the new password directly to the client or user, so there's no need to relay it manually.

## Improvements & Fixes

### Fixed: Attaching a GUTMA Flight Log Didn't Update the Mission

Attaching a FlytBase/dock flight to a mission from the **Mission Log** tab uploaded the raw GUTMA file, but never read it — the mission's actual start/end time, flight duration, distance flown, and battery charge stayed empty, and the Post Flight tab showed nothing from the imported flight.

- **Auto-Sync on Attach** — Attaching a flight now parses the GUTMA log immediately and writes actual start, actual end, flight duration, distance flown, and battery charge start/end onto the mission, so the Mission Log and Post Flight tabs reflect the import right away — no separate sync step needed.
- **Missing End Time Fixed** — Some GUTMA exports only report elapsed seconds per data point rather than an absolute end time. When no explicit end time is present, it's now derived from the logging start time plus the last recorded timestamp, instead of being left blank.

### Fixed: Deleting a Client Left Behind a Login Account

Deleting a client previously removed only the client record itself — the portal login it created (and its Supabase Auth account) was left behind. That orphaned account then blocked re-creating a new client, or a user, with the same email or username, since the system still considered it taken.

Deleting a client now also removes its portal login and Supabase Auth account, while correctly unassigning (rather than deleting) any personnel who were simply scoped to that client. The email and username are fully released, so they can immediately be reused when creating a new client.

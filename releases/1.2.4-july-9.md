## What's New

### Attach a Flight Log While Creating a New Mission

From Control Center → Flights, choosing "Create New Mission" for a flight that has no matching existing mission now creates the mission and immediately attaches that flight's log in one step — no more coming back afterward to attach it manually. The new mission form pre-fills the flight's start time, end time, and distance flown, and the mission is created with status **Completed**, since it represents a flight that's already happened. If the drone you select doesn't share the same serial number as the flight log, you'll see a warning so you can catch a mismatch before saving.

### Editable Mission Code on ccPlatform Import

The "Import Mission" wizard now shows an editable **Mission Code** field — previously the code was always auto-generated from the log with no way to override it. The wizard also displays the drone serial number detected from the uploaded (or selected) flight log, so you can confirm it matches the drone you're assigning before importing.

## Improvements & Fixes

### Fixed: Imported Missions Missing Start/End Times

Missions imported from a ccPlatform flight log were never getting their actual start time, end time, or flight distance — the import parser was reading the wrong part of the GUTMA file structure, so these fields silently stayed empty. Imported missions now correctly show start and end time like any other logged mission, with duration and distance calculated from the flight path even when the log itself doesn't report an explicit end time.

### Fixed: Imported Missions Not Appearing in the Operations Table

Imported missions weren't being linked to the client selected during import, and the Operations table was sorted by scheduled/flight date rather than by when the mission was added — so an import for a past flight could be silently pushed off the first page, or excluded when filtering by client. Imports are now correctly linked to their client, and the table shows the newest missions first by default.

### Fixed: Flight Log Attachment Failing Across Organizations

Attaching a Control Center flight log to a mission — from the Flights page, a mission's Logs tab, or Mission Complete — always used your default FlytBase credentials instead of the ones for the organization the flight was actually browsed under. In any multi-organization setup this made the attach silently fail. Attaching a log now always resolves the correct organization's credentials.

### Fixed: Serial Number Mismatch Blocked Flight Log Attachment

If a flight log's aircraft serial number didn't exactly match the drone assigned to a mission, attaching the log used to fail outright with no way to override it. The log is now always attached; a mismatch shows a warning naming both serial numbers instead of a hard failure.

### Fixed: New Missions Always Showed as "Scheduled"

A status-name mismatch meant every mission created through the New Mission form was internally recorded with "Scheduled" status regardless of what was actually selected, throwing off status-based views like the daily board. New missions now get the correct internal status from the start.

### Fixed: Manual Flight Log Uploads Didn't Pre-Fill Post-Flight Data

Attaching a Control Center flight automatically filled in actual start/end time, duration, distance, and battery charge — but manually uploading a `.gutma`/`.zip` log file did not. Manual uploads now sync the same post-flight fields automatically.

### Fixed: Duplicate Map on the Post Flight Tab

The Post Flight tab was rendering the flight path map twice. It now shows once, in its labeled "Flight Path" section.
